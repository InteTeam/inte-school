## The 9-Step Feature Process

### ⚠️ Step 0 — Read Guidelines FIRST (Mandatory, no exceptions)

**Do not write a single line of code or documentation without completing this step.**

1. Read `CLAUDE.md` (always — it may have been updated)
2. Read `docs/WORKFLOW_ENFORCEMENT.md`
3. Read `docs/FEATURE_DESIGN_CHECKLIST.md`
4. Read `docs/DOCUMENTATION_STANDARDS.md`
5. Before any migration → read `docs/DATABASE_CONVENTIONS.md`
6. Before any frontend component → read `docs/COMPONENT_REUSE_CHECKLIST.md`
7. For performance/scale decisions → read `docs/planning/ARCHITECTURE.md`

**You must be able to answer before proceeding:**
- [ ] What is the exact flash message format?
- [ ] Where does `#[UsePolicy]` go — model or controller?
- [ ] What syntax for validation rules — array or pipe?
- [ ] What test framework — PHPUnit or Pest?
- [ ] What is the multi-tenancy trait and what does it do?
- [ ] What is the graceful fallback for this feature?
- [ ] What values are configurable via JSONB settings vs hardcoded?

**⛔ Cannot answer all of the above → re-read, do not proceed.**

---

### Step 1 — Define Business Requirements

**Output:** `docs/features/{feature_name}/README.md`

1. Create the feature directory:
   ```bash
   mkdir -p docs/features/{feature_name}
   ```

2. Fill in the README using the template from `docs/DOCUMENTATION_STANDARDS.md`:
   - [ ] Overview — what and why
   - [ ] User stories with acceptance criteria
   - [ ] Business requirements
   - [ ] Success metrics
   - [ ] Which roles are affected
   - [ ] **Graceful fallback** — what happens when this feature's primary path fails?
   - [ ] **Flexible values** — which settings belong in JSONB rather than hardcoded?
   - [ ] Copy and start ticking the `docs/FEATURE_DESIGN_CHECKLIST.md` into this README

---

### Step 2 — Design Technical Architecture

**Output:** `docs/features/{feature_name}/architecture.md`

1. Identify all affected components:
   - [ ] Tables needed (cross-reference `docs/planning/ARCHITECTURE.md` schema first — the table may already be designed)
   - [ ] Models (does `HasSchoolScope` + `HasUlids` apply? Almost certainly yes)
   - [ ] Services (domain-organised, `final`, <250 lines)
   - [ ] Controllers (<150 lines)
   - [ ] Policies (`#[UsePolicy]` on model)
   - [ ] Observers (does this feature need cache invalidation?)
   - [ ] Jobs (anything >5s must be queued; which queue lane — high/default/low?)
   - [ ] Routes (which route file does this belong in?)
   - [ ] Frontend pages + components

2. Architecture constraints checklist:
   - [ ] Multi-tenant: every query goes through `HasSchoolScope`
   - [ ] PostgreSQL: JSONB for JSON, `VECTOR(768)` for embeddings if relevant
   - [ ] Indexes planned: `idx_school_created` mandatory, `idx_school_status` if status column
   - [ ] Caching strategy documented (what is cached, TTL, which Observer clears it)
   - [ ] Queue lane assigned for any async jobs
   - [ ] **Fallback path documented** — what happens on failure at each step

3. Critical questions before proceeding:
   - Does this touch the notification cascade? → Verify graceful fallback chain
   - Does this add a school setting? → Store in JSONB, no UI until needed
   - Does this require a new calendar type, message type, or task type? → Use string, not enum
   - Does this have an external API dependency? → Document the failure mode

---

### Step 3 — Plan Database Changes

**Output:** `docs/database/migrations/{next_number}_{name}.md`

1. Find the next migration number:
   ```bash
   ls docs/database/migrations/ | sort -n | tail -1
   ```

2. Create the migration doc using the template from `docs/DOCUMENTATION_STANDARDS.md`

3. Document:
   - [ ] All columns with types (use `JSONB` not `JSON`)
   - [ ] Mandatory indexes: `idx_school_created (school_id, created_at)`, `idx_school_status` if applicable
   - [ ] Foreign keys with explicit CASCADE rule
   - [ ] Relationships
   - [ ] Model configuration (`$fillable`, `casts()`, traits)
   - [ ] PGVector: if embedding column, specify `VECTOR(768)` + IVFFlat index

4. **PostgreSQL conventions (non-negotiable):**
   - JSON columns → `JSONB` always
   - Check migration references `docs/DATABASE_CONVENTIONS.md` before writing
   - `CREATE EXTENSION IF NOT EXISTS vector;` already handled in migration 001 — do not repeat

---

### Step 4 — Inventory Frontend Components

**Output:** `docs/features/{feature_name}/components.md`

**⚠️ INVENTORY EXISTING COMPONENTS FIRST. Never plan a component that already exists.**

1. Before planning any component, list existing ones:
   ```bash
   # Check existing components
   ls resources/js/Components/ui/
   ls resources/js/Components/Atoms/
   ls resources/js/Components/Molecules/
   ls resources/js/Components/Organisms/
   ls resources/js/Pages/{relevant_module}/
   ```

2. Create `docs/features/{feature_name}/components.md` with two sections:
   - **Reuse:** existing component, import path, props needed
   - **New:** component name, purpose, why it can't be reused, props interface

3. DRY enforcement:
   - ❌ Never create `Button.tsx` if `ui/button.tsx` exists
   - ❌ Never create `Input.tsx` if `ui/input.tsx` exists
   - ✅ Always list existing before planning new
   - ✅ Document exact import paths for reused components

4. Role context: which layout wraps this page? (`AuthLayout`, `SchoolLayout`, `ParentLayout`)

**⛔ Planned any component without checking existing → redo the inventory.**

---

### Step 5 — Choose Implementation Phase

| Phase | When | What |
|---|---|---|
| **UI Demo** | Stakeholder review | Mock data, skeleton auth, rapid iteration |
| **Integration** | Feature testing with real data | Real DB, basic auth, school scoping enabled |
| **Production** | Live deployment | Full auth + policies, caching, queue jobs active |

Match the phase to current project stage. Do not over-engineer a UI Demo into Production mode prematurely.

---

### Step 6 — Write Tests First (TDD)

**Output:** Failing test file in `tests/Feature/{Module}/`

1. Create test file:
   ```bash
   docker compose exec php-fpm php artisan make:test {Module}/{Feature}Test --no-interaction
   # Convert to: final class {Feature}Test extends TestCase { use RefreshDatabase; }
   ```

2. Write failing tests covering:
   - [ ] Guest access (should redirect/401)
   - [ ] Wrong role (should 403)
   - [ ] Correct role (should succeed)
   - [ ] Multi-tenant isolation (school A cannot access school B data)
   - [ ] Valid input → expected outcome
   - [ ] Invalid input → validation errors
   - [ ] Business logic edge cases
   - [ ] Graceful fallback path (e.g., mock service failure → fallback fires)

3. Run to confirm they fail:
   ```bash
   docker compose exec php-fpm php artisan test --filter={Feature}Test
   ```

4. **PHPUnit only — never Pest.** `final class`, `use RefreshDatabase`, factories for data.

### Step 7 — Implement

**Order of implementation:**

```bash
# 1. Migration
docker compose exec php-fpm php artisan make:migration {name} --no-interaction

# 2. Model (with factory)
docker compose exec php-fpm php artisan make:model {Model} --factory --no-interaction
# Add: HasSchoolScope, HasUlids, declare(strict_types=1)

# 3. Policy
docker compose exec php-fpm php artisan make:policy {Model}Policy --no-interaction
# Add #[UsePolicy] to model

# 4. Service (create manually — artisan make:service doesn't exist)
# final class, constructor injection, <250 lines

# 5. Observer (if caching involved)
docker compose exec php-fpm php artisan make:observer {Model}Observer --no-interaction

# 6. Job (if async work)
docker compose exec php-fpm php artisan make:job {Name}Job --no-interaction
# Assign correct queue lane in handle()

# 7. Form Request
docker compose exec php-fpm php artisan make:request {Name}Request --no-interaction

# 8. Controller
docker compose exec php-fpm php artisan make:controller {Name}Controller --no-interaction
# <150 lines, no #[UsePolicy], use auth()->user()->can() + abort(403)

# 9. Route (add to correct domain route file)

# 10. Frontend (React/TypeScript pages + components)
```

**During implementation:**
- [ ] Every PHP file starts with `declare(strict_types=1);`
- [ ] Validation uses array syntax, not pipe strings
- [ ] JSON columns cast as `array` in model `casts()`
- [ ] File upload routes use `Route::match(['PUT', 'POST'], ...)`
- [ ] File upload forms use `router.post()` on frontend

## Debugging Procedure

When something breaks, follow this order — do not skip steps:

1. **Read the error message fully** before touching code
2. **Check logs:** `storage/logs/laravel-$(date +%Y-%m-%d).log`
3. **Check queue worker logs** if the issue involves async jobs:
   ```bash
   docker compose logs queue-worker --tail=50
   ```
4. **Narrow scope** — reproduce in the smallest possible test case
5. **Surgical fix** — fix the specific cause, do not refactor surrounding code
6. **Re-run tests** after fix to confirm nothing regressed
7. If blocked after 20 minutes → document what you've tried, ask for input

**Never:**
- Retry the same failing action repeatedly without changing something
- Delete a lock file without investigating what holds it
- Use `--no-verify` to bypass hooks
- Force push without explicit instruction

## Code Review Checklist

Before any commit:

**PHP:**
- [ ] `declare(strict_types=1);` on all new files
- [ ] `HasSchoolScope` + `HasUlids` on all new tenant models
- [ ] `#[UsePolicy]` on model, not controller
- [ ] Validation uses array syntax
- [ ] No hardcoded values that should be in JSONB settings
- [ ] Services are `final`, <250 lines
- [ ] Controllers <150 lines
- [ ] Heavy operations (>5s) are queued
- [ ] `school_id` not in `$fillable` on any model
- [ ] All user-facing list queries use `orderBy('created_at', 'desc')`

**Security:**
- [ ] New endpoints have rate limits defined in `bootstrap/app.php`
- [ ] File uploads validated by MIME type (not extension only)
- [ ] CSV import/export fields sanitized against formula injection
- [ ] Student data features default to most private option (UK Children's Code)
- [ ] Any new token/key stored hashed in DB

**PostgreSQL:**
- [ ] JSON columns use `JSONB`
- [ ] Mandatory indexes present
- [ ] Foreign key cascade rules explicit

**Tests:**
- [ ] PHPUnit only — `final class extends TestCase`, `use RefreshDatabase`
- [ ] Multi-tenant isolation tested
- [ ] Graceful fallback path tested

**Frontend:**
- [ ] All props interfaces defined — no `any`
- [ ] File upload forms use `router.post()`
- [ ] No components created that already exist in `ui/` or `Atoms/`

**Principles:**
- [ ] Graceful fallback implemented and tested
- [ ] Configurable values stored in JSONB, not hardcoded
- [ ] Docs complete
